RATE LIMIT ERROR FORMAT
-----------------------
- HTTP status: 429
- Error code string: "RESOURCE_EXHAUSTED" (Google's own error code)
- Via OpenAI-compatible endpoint: raises openai.RateLimitError with HTTP 429
- Error message typically contains: "429", "RESOURCE_EXHAUSTED", "resource exhausted", "quota"
- Example message: "Error code: 429 - {'error': {'code': 429, 'message': 'RESOURCE_EXHAUSTED: ...'}}"

CURRENT DETECT.PY RATE-LIMIT DETECTION (line 267):
  is_rate_limit = "429" in last_exc_str or "rate limit" in exc_str_lower or "rate_limit" in exc_str_lower

COVERAGE ANALYSIS:
- "429" in last_exc_str: YES — catches Gemini 429 RESOURCE_EXHAUSTED (HTTP 429 is in the error string)
- "rate limit" / "rate_limit": catches OpenAI-style messages
- MISSING: "resource_exhausted" / "RESOURCE_EXHAUSTED" — Gemini-specific error code
  → If the proxy surfaces "RESOURCE_EXHAUSTED" without "429" in the string, it would fall through
    to the generic 1s/2s/4s backoff instead of the 4s/8s/16s rate-limit backoff
  → HOWEVER: via OpenAI-compatible endpoint, the HTTP status IS 429, so "429" will always be
    in the exception string from the openai library (openai.RateLimitError always includes HTTP 429)

CONCLUSION: Current detection is adequate for the OpenAI-compatible Gemini endpoint.
The "429" string check catches all Gemini rate limits surfaced via the openai library.
Adding "resource_exhausted" as a belt-and-suspenders check is recommended.
